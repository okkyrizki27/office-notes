Bagian 2

Latar Belakang & Gap Saat Ini

Kondisi PM Shutdown saat ini dan permasalahan yang ingin diselesaikan

Kondisi Saat Ini

PM Shutdown saat ini hanya menghandle backlog execution — yaitu temuan hasil inspeksi yang dimasukkan ke dalam scheduled service untuk perbaikan. Tidak ada dokumentasi terstruktur tentang pekerjaan yang dilakukan selama service berlangsung.

Tab Form di workcard PM Shutdown mobile sudah ada dalam aplikasi namun masih tersembunyi (hidden) dan belum diimplementasi. Phase 1 ini akan mengaktifkan dan mengisi tab tersebut.

Gap yang Ditemukan

Di industri heavy equipment mining, scheduled service berbasis periodical service hour: 250 hrs, 500 hrs, 1000 hrs, 2000 hrs. Dalam kondisi aktual di lapangan sebelum digitalisasi, setiap service memiliki bundle form yang harus diisi — namun ini belum terakomodasi di Digiman+.

Jenis pekerjaan yang dicatatBacklog execution sajaBacklog + Service form bundle
Dokumentasi serviceTidak adaService sheet + form tambahan sesuai kebutuhan
Pengisian formSatu user, satu formBanyak mechanic, satu form, offline-first
Tracking aktivitas mechanicTidak adaRekaman masuk/keluar form per shift
Tab Form di workcardHidden, belum implementasiAktif, menampilkan bundle form

Bagian 4

Fitur 1: Service Bundle Planning

Planner menyusun bundle form untuk sebuah scheduled service — dilakukan di web (Digiplan)

Konsep Service Bundle

Untuk setiap service execution, ada kumpulan form (bundle) yang harus diisi oleh mechanic di lapangan. Bundle ini terdiri dari form mandatory dan optional, ditentukan oleh Planner.

Jenis Form dalam Bundle

Jenis FormContohStatus DefaultSiapa yang Tentukan
Service SheetChecklist komponen, pengukuranMandatorySistem (suggestion), Planner bisa ubah
Form PengelasanWelding work formOptionalPlanner
Form Tyre R&IPelepasan & pemasangan tyreOptionalPlanner
Return to Operation (RTO)Form serah terima unitOptionalPlanner
Form lainnyaSesuai kebutuhan serviceOptionalPlanner

Keputusan desain: Service sheet di-suggest otomatis oleh sistem berdasarkan Equipment Model × Service Hour, dengan status default Mandatory. Namun authority penuh ada di Planner — sistem hanya memberi suggestion, Planner dapat mengubah mandatory/optional untuk semua form.

Peran Planner dalam Menyusun Bundle

Planner adalah pemilik scope pekerjaan. Planner mengetahui: unit apa yang di-service, interval berapa (250/500/1000/2000 hrs), pekerjaan apa yang akan dilakukan, dan durasi service. Dari informasi ini, Planner menentukan komposisi bundle form.

Filter Form Suggestion oleh Sistem

Sistem menampilkan suggestion form berdasarkan dua kriteria sekaligus:

  • Form yang di-mapping ke model equipment unit yang di-service
  • Form yang di-mapping ke model equipment + service hour yang bersangkutan

Planner juga dapat melakukan pencarian manual untuk menemukan form yang tidak muncul dalam suggestion.

Alur: Planner Menyusun Bundle

Diagram 4.1 — Alur Service Bundle Planning (Planner)
flowchart TD A([Planner buka Scheduled Service Plan]) --> B[Pilih unit & interval service\n250/500/1000/2000 hrs] B --> C{Service sheet tersedia\ndi Form Builder?} C -->|Ya| D[Sistem auto-select Service Sheet\nIsMandatory = true] C -->|Tidak| E[Tidak ada blocker\nLanjut tanpa service sheet] D --> F[Sistem tampilkan suggestion form\nberdasarkan model & model+service hour] E --> F F --> G[Planner review, tambah / cari form lain] G --> H[Planner set tiap form: Mandatory atau Optional] H --> I[⚠ Sistem tampilkan WARNING\nForm tidak bisa di-unassign setelah disimpan] I --> J{Planner konfirmasi Save?} J -->|Tidak| G J -->|Ya| K[Sistem buat Task per form - SQL\n+ copy template JSON - Cosmos] K --> L([✓ Bundle siap untuk eksekusi di lapangan])

Aturan Penting: Bundle Tidak Bisa Diubah Setelah Disimpan

Untuk MVP Phase 1: form yang sudah di-assign ke bundle tidak bisa dihapus setelah Planner menyimpan. Sistem akan menampilkan peringatan (warning) sebelum Planner mengkonfirmasi penyimpanan. Planner masih bisa menambah form setelah save pertama, tapi tidak bisa menghapus.

Aturan jika Service Sheet Tidak Tersedia

Jika service sheet untuk kombinasi model equipment × service hour belum dibuat di Form Builder, tidak ada blocker — Planner tetap dapat menyimpan bundle dan eksekusi tetap dapat berjalan sampai selesai. Prinsip: konsistensi di seluruh alur.

Validasi Finish Execution

Sebelum Supervisor/Foreman dapat melakukan Finish Execution untuk menandai WorkOrder selesai, sistem memvalidasi dua hal:

ValidasiSebelum Phase 1Setelah Phase 1
Backlog execution selesai✅ Ada✅ Tetap ada
Semua mandatory form status Complete❌ Tidak ada✅ Ditambahkan

Optional form tidak memblokir Finish Execution — hanya mandatory form yang menjadi syarat.

Finish Execution bisa dilakukan offline — selama data di device lokal menunjukkan semua mandatory form sudah Complete dan backlog execution selesai. Prinsip offline-first berlaku penuh untuk semua aksi termasuk Finish Execution.

Bagian 6

Fitur 3: Multi-Mechanic Form Execution

Satu form dapat diisi oleh beberapa mechanic sekaligus dengan dukungan offline-first penuh

Prinsip Dasar

  • Satu form dalam bundle dapat diisi oleh beberapa mechanic sekaligus
  • Semua mechanic menggunakan mode offline-first — tidak perlu koneksi internet saat mengisi
  • Pembagian kerja antar mechanic dilakukan secara operasional (oleh Supervisor, Foreman, atau mechanic sendiri) — sistem tidak mengatur siapa mengisi bagian mana
  • Submission bersifat per form, bukan per bundle

Alur Lengkap: Dari Buka Form hingga Finish Execution

Diagram 6.1 — Alur Form Execution (Multi-Mechanic)
flowchart TD A([Mechanic buka form di workcard]) --> B[Baca safety precautions] B --> C[Klik Mulai\nSistem buat TaskPersonalizedLog record] C --> D[Mechanic isi field-field form\nData tersimpan lokal otomatis - auto-save] D --> E{Ada koneksi internet?} E -->|Ya| F[Auto-sync: upload dirty fields\n+ download update dari rekan] E -->|Tidak| G[Tetap isi form offline\nSemua tersimpan lokal] F --> H{Semua mandatory\nfields terisi?} G --> H H -->|Belum| D H -->|Ya| I[Review form via Summary] I --> J[Klik Submit] J --> K[Form status: Complete\nMasuk approval workflow] K --> L{Semua mandatory form\ndalam bundle Complete?} L -->|Belum| M[Mechanic lain lanjut\nisi form masing-masing] L -->|Ya| N[Supervisor/Foreman\nklik Finish Execution] N --> O([✓ WorkOrder Complete])

Visibility Antar Mechanic

Mechanic dapat saling melihat progress pengisian hingga level field per field — bukan hanya per tab. Mechanic B dapat melihat field mana saja yang sudah diisi oleh Mechanic A beserta identitasnya.

Mekanisme visibilitas: Auto-sync aktif saat ada koneksi — debounce 3 detik setelah input (upload + download) dan background timer 30 detik (download saja). Near real-time tanpa persistent connection. Saat offline, update terlihat hanya setelah sync.

Aturan Submit

AspekAturan
Siapa yang submitSatu mechanic sebagai perwakilan per form
Pre-conditionSemua mandatory fields terisi di device submitter
Submit offline✅ Bisa — di-queue, dikirim ke server saat ada koneksi
Review sebelum submitTersedia fitur Summary untuk review form sebelum submit
Duplicate submissionFirst submit wins by timestamp — submit kedua ditolak

Penanganan Konflik Antar Mechanic

Karena beberapa mechanic bisa mengisi form yang sama secara offline, ada kemungkinan dua mechanic mengisi field yang sama. Aturannya:

First Write Wins: Field yang diisi lebih awal (berdasarkan timestamp perangkat saat input) yang dipertahankan. Jika ada field yang tidak tersimpan karena sudah diisi lebih awal oleh mechanic lain, sync tetap dianggap berhasil dan mechanic mendapat notifikasi yang jelas tentang field mana yang tidak tersimpan.

Edge Cases Penting

SituasiPenanganan
Auto-save & crash perangkatSudah diimplementasi — data tersimpan lokal otomatis
WorkOrder dibatalkan saat form diisiTidak bisa terjadi — jika plan sudah start eksekusi, Planner tidak bisa cancel
Mechanic sakit/pergi, data belum sync — form contentJika sync terjadi setelah form sudah Complete, data ditolak
Mechanic sakit/pergi, data belum sync — activity recordsTetap diterima selama StartDate < waktu submit (activity adalah historical log)

Fitur 4: Identitas & Autentikasi Mechanic

Mechanic diidentifikasi menggunakan akun individual — email / UPN

Keputusan

Mechanic menggunakan akun individual — login dengan email / UPN masing-masing. Shared user tidak dipilih.

Dasar Keputusan

Semua mechanic sudah terdaftar sebagai karyawan di tabel UserEmploymentProfile. Kolom UserCode (email) bersifat not null untuk semua mechanic — fondasi akun individual sudah tersedia, tinggal provisioning.

KolomKeteranganCatatan
UserCodeEmail / UPN karyawanNot null — digunakan sebagai mechanic identity
EmployeeIdID unik karyawanUnique identifier
SiteIdSite lokasi kerjaFilter per site

Implikasi

  • Mechanic login langsung dengan email/UPN — tidak perlu pilih nama dari daftar
  • Identity di form field (updated_by) dan activity log (TaskPersonalizedLog) menggunakan UserCode
  • Audit trail solid — setiap aksi tercatat atas nama mechanic yang sebenarnya
  • Tidak perlu konfigurasi mapping tambahan

Fitur 5: Mechanic Activity Tracking

Pencatatan aktivitas masuk/keluar form per mechanic per shift untuk kebutuhan reporting

Konsep

Setiap kali mechanic masuk dan keluar dari form, aktivitasnya direkam. Satu mechanic bisa punya beberapa sesi dalam satu form jika bekerja di shift berbeda atau hari berbeda.

AksiKeterangan
Mulai (Start)Mechanic memulai pengerjaan form — klik tombol "Mulai" setelah membaca safety precautions
Selesai (Finish/Close)Mechanic keluar dari form sementara — bisa kembali lagi nanti
SubmitForm selesai secara final — tidak bisa kembali mengisi

Retry Mechanism

Digiman+ sudah memiliki retry mechanism. Ditambahkan prinsip idempotent batch:

  • Setiap sync batch diberi unique requestId
  • Jika partial failure → retry batch yang sama
  • Server menggunakan fieldId + timestamp sebagai idempotency key
  • Jika dua mechanic memiliki timestamp identik pada field yang sama: sync pertama yang tiba diproses, sync berikutnya dianggap berhasil (no-op)

Conflict Resolution

Bagaimana konflik data ditangani saat beberapa mechanic mengisi form yang sama secara offline

Strategi: Optimistic Locking + First Write Wins

Tidak ada lock saat field dibuka. Conflict dideteksi saat sync berdasarkan device timestamp saat input.

-- Per-field data structure di Cosmos:
field: {
  value     : any,
  timestamp : DateTime,   -- device clock saat input (known limitation: bisa tidak akurat)
  updated_by: string      -- UserCode mechanic
}

-- Conflict resolution:
IF existing_field.timestamp < incoming_field.timestamp:
  DISCARD incoming_field (existing wins)
  REPORT field sebagai "tidak tersimpan" ke client
ELSE IF existing_field.timestamp == incoming_field.timestamp:
  DISCARD incoming_field (no-op, dianggap berhasil)
ELSE:
  UPDATE field dengan incoming value

Known Limitation: Device Clock

Timestamp menggunakan device clock — bisa tidak akurat jika perangkat tidak tersinkronisasi. Ini adalah acceptable risk yang sudah didokumentasikan. Di lapangan tambang, kemungkinan dua mechanic mengisi field yang sama di momen yang benar-benar identik sangat kecil.

Activity Tracking — Detail Teknis

Implementasi auto-close mechanism dan shift detection

Auto-Close Mechanism — Client-Side

Auto-close adalah mekanisme baru yang dibangun di client. Server hanya menerima dan menyimpan data — tidak ada deteksi auto-close di server side.

-- Saat mechanic klik Mulai:
function handleStartClick(taskPersonalizedId, deviceTimestamp, siteCode):
  currentShift = getShift(deviceTimestamp, siteCode)

  // 1. Cek deduplication: open session di shift yang sama
  openSameShift = TaskPersonalizedLog.find(
    taskPersonalizedId, currentShift.Name, EndDate = null
  )
  if openSameShift exists: return  // tidak buat record baru, langsung masuk form

  // 2. Auto-close open session dari shift berbeda
  openOtherShift = TaskPersonalizedLog.find(taskPersonalizedId, EndDate = null)
  if openOtherShift exists:
    prevShift = getShift(openOtherShift.StartDate, siteCode)
    openOtherShift.EndDate = calculateShiftEnd(openOtherShift.StartDate, prevShift)
    queue UPDATE openOtherShift

  // 3. Buat sesi baru
  queue INSERT TaskPersonalizedLog (taskPersonalizedId, StartDate=deviceTimestamp, ShiftName=currentShift.Name)

-- calculateShiftEnd:
function calculateShiftEnd(startDate, shift):
  if shift.EndShift < shift.StartShift:  // crosses midnight
    return datetime(startDate.date + 1 day, shift.EndShift)
  else:
    return datetime(startDate.date, shift.EndShift)

-- Auto-close via Submit:
function handleFormSubmit(taskId, submitTime, siteCode):
  openSessions = TaskPersonalizedLog.findAll(taskId, EndDate = null)
  for session in openSessions:
    shift = getShift(session.StartDate, siteCode)
    shiftEnd = calculateShiftEnd(session.StartDate, shift)
    session.EndDate = min(shiftEnd, submitTime)
    queue UPDATE session

Shift Detection dari Timestamp

-- Logika penentuan shift:
function getShift(timestamp, siteCode):
  time = timestamp.time
  shifts = SiteShift.where(SiteCode = siteCode, IsActive = true)
  for shift in shifts:
    if shift.EndShift > shift.StartShift:  // same day
      if shift.StartShift <= time < shift.EndShift: return shift
    else:  // crosses midnight
      if time >= shift.StartShift OR time < shift.EndShift: return shift

-- Contoh untuk site 2009:
-- 06:00–17:59 → Day Shift (EndShift 18:00 > StartShift 06:00)
-- 18:00–05:59 → Night Shift (EndShift 06:00 < StartShift 18:00)

1. Overview

API ini menyediakan akses read-only ke data hasil pengisian form IIR (Inspection Report) di Digiman+, model aksesnya adalah pull/polling — sistem eksternal yang menarik data secara berkala (bukan webhook/push dari Digiman+).

Setiap submission mencakup metadata (site, equipment, waktu submit) dan seluruh jawaban form (termasuk foto, terstruktur per tab & section).

Base URL

https://{host}/api/v1

Placeholder — domain final (staging/production) akan diinformasikan terpisah sebelum go-live.

2. Autentikasi

Autentikasi pakai Client ID + Secret Key, di-exchange ke Bearer token lewat OAuth2 Client Credentials Grant — bukan mengirim Client ID/Secret langsung di setiap request data.

2.1 Exchange Token

Request
POST /oauth/token
Content-Type: application/x-www-form-urlencoded

grant_type=client_credentials&client_id={CLIENT_ID}&client_secret={CLIENT_SECRET}
Response
{
  "access_token": "eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9...",
  "token_type": "Bearer",
  "expires_in": 3600
}

2.2 Pakai Token

Sertakan token di header Authorization pada setiap request ke endpoint data:

Authorization: Bearer {access_token}
Penting
Token punya masa berlaku terbatas (expires_in, contoh: 1 jam). Setelah expired, lakukan exchange ulang untuk dapat token baru — jangan simpan/pakai ulang client_secret di request data.

2.3 Scope Akses

Client ID yang valid otomatis punya akses ke semua site — tidak ada pembatasan akses per site. Parameter siteCode/equipmentNumber pada request murni untuk menyaring hasil pencarian, bukan kontrol keamanan.

4. Response Data Model

Struktur lengkap response (baik dari endpoint list maupun single-fetch):

{
  dateFrom, dateTo,              // hanya di endpoint list — echo balik parameter request
  totalRecords, page, pageSize, hasMore,   // hanya di endpoint list
  submissions: [                 // hanya di endpoint list — endpoint single-fetch return submission langsung
    {
      formSubmissionId, formCode, formVersion, formName,
      siteCode, equipmentNumber,
      submittedBy, submittedAt, syncAt,
      tabs: [
        {
          title,
          sections: [
            {
              title,
              answers: [
                { taskCode?, label, value, lastUpdatedBy, photos: [{ url }], number?, remark? }
              ]
            }
          ]
        }
      ]
    }
  ]
}

Catatan bentuk value
  • Jawaban teks/pilihan biasa (mis. hasil pemeriksaan "Ok"/"Not Ok", teks bebas) → string biasa
  • Jawaban foto tunggal (satu foto per pertanyaan) → value berisi identifier internal foto tersebut (bisa diabaikan — pakai photos[].url untuk link foto aktualnya)
  • Jawaban multi-foto (beberapa foto per pertanyaan) → value = null, semua foto ada di photos[]
  • Jawaban tanda tangan → value berisi gambar dalam format data:image/png;base64,... langsung (bukan URL) — bisa langsung dipakai sebagai src elemen <img>

4.5 Objek Photo

Wajib di-handle
Kalau photos[].url bernilai null, artinya foto tersebut ada tapi gagal di-resolve saat request diproses. Jangan asumsikan selalu berisi URL valid — tampilkan placeholder/skip, jangan sampai integrasi gagal total karena satu foto null.

5. Pagination

Endpoint list (3.1) selalu dipaginasi. Default pageSize adalah 20, maksimal 100.

Loop paginasi standar:

page = 1
loop:
  response = GET /iir-form-submissions?dateFrom=...&dateTo=...&page={page}&pageSize=100
  process(response.submissions)
  if response.hasMore == false:
    break
  page += 1

totalRecords merepresentasikan total submission yang match filter di semua halaman, bukan cuma halaman berjalan.

6. Filtering & Panduan Polling

6.1 Rentang Tanggal — Maksimal 7 Hari

dateFrom/dateTo wajib diisi, dan jarak antara keduanya maksimal 7 hari. Kalau butuh rentang lebih panjang, pecah jadi beberapa request berurutan.

6.2 submittedAt vs syncAt — Kenapa Bisa Beda

Digiman+ mendukung pengisian form secara offline di lapangan. Karena itu ada 2 timestamp dengan makna berbeda:

  • submittedAt — waktu inspector menyelesaikan form di device. Bisa terjadi saat device offline.
  • syncAt — waktu data tersebut benar-benar tersedia di server (setelah device online dan sync). Bisa lebih lambat dari submittedAt — kadang beda beberapa menit, kadang beda beberapa hari kalau device lama offline.
Implikasi Penting
Filter dateFrom/dateTo pada endpoint list berdasarkan syncAt, bukan submittedAt. Jadi wajar kalau suatu submission dengan submittedAt beberapa hari yang lalu baru muncul di hasil polling hari ini — itu karena device-nya baru online dan sync hari ini. Desain ini memastikan sistem eksternal tidak pernah kehilangan data akibat keterlambatan sync, asalkan window polling mencakup waktu syncAt-nya.

Rekomendasi polling: jalankan polling secara berurutan tanpa celah (mis. window kemarin berakhir jam 23:59:59, window hari ini mulai jam 00:00:00) — cukup andalkan syncAt, tidak perlu overlap manual di sisi sistem eksternal.

6.3 Filter Site & Equipment

siteCode dan equipmentNumber bersifat opsional dan bisa dikombinasikan dengan filter tanggal. Kedua field ini juga selalu muncul di response (lihat 4.2) sehingga caller bisa mengelompokkan hasil per site/equipment tanpa perlu filter di request.

7. Rate Limiting

Rate limit berlaku per Client ID (bukan per IP):

WindowLimit
Per menit30 request
Per hari2.500 request

Limit ini cukup longgar untuk pola polling berkala maupun paginasi banyak halaman — kira-kira 1 request tiap 2 detik.

Saat Limit Terlampaui

HTTP/1.1 429 Too Many Requests
Retry-After: 12

{
  "error": {
    "code": "RATE_LIMITED",
    "message": "Rate limit exceeded, retry after 12 seconds",
    "details": null
  }
}

Gunakan header Retry-After (dalam detik) untuk menentukan kapan retry, dan monitor header X-RateLimit-Remaining di setiap response supaya bisa self-throttle sebelum kena limit.

8. Error Handling

Semua error mengikuti format envelope yang sama:

{
  "error": {
    "code": "DATE_RANGE_TOO_LARGE",
    "message": "Date range exceeds maximum of 7 days",
    "details": null
  }
}

code adalah string machine-readable — gunakan ini untuk branching logic, jangan parsing message (yang human-readable dan bisa berubah). details nullable, dipakai untuk kasus validasi dengan lebih dari satu field bermasalah (array of {field, reason}).

HTTP StatuscodeKapan Terjadi
400INVALID_REQUESTParameter malformed (mis. format tanggal salah)
400DATE_RANGE_TOO_LARGEdateTo - dateFrom lebih dari 7 hari
400INVALID_PAGE_SIZEpageSize lebih dari 100
401UNAUTHORIZEDToken tidak ada / invalid / expired
403FORBIDDENClient valid tapi tidak punya akses ke resource yang diminta
404NOT_FOUND(single-fetch) formSubmissionId tidak ditemukan
429RATE_LIMITEDLihat [Rate Limiting](#sec-7)
500INTERNAL_ERRORUnexpected server error

9. Best Practices Integrasi

  • Selalu loop sampai hasMore: false — jangan asumsikan semua data selalu muat di 1 halaman.
  • Handle photos[].url: null dengan graceful — jangan sampai 1 foto gagal resolve membuat seluruh proses integrasi gagal.
  • Simpan formSubmissionId + syncAt sebagai watermark untuk tracking data yang sudah diproses, memudahkan dedup kalau ada request yang di-retry.
  • GET bersifat idempotent — aman untuk retry otomatis kalau request gagal karena masalah jaringan (bukan karena error 4xx).
  • Terapkan exponential backoff saat menerima 429, hormati header Retry-After.
  • Gunakan endpoint single-fetch (3.2) untuk retry granular — kalau 1 submission gagal diproses dari hasil polling, tidak perlu re-poll seluruh rentang tanggal, cukup fetch ulang submission tersebut by ID.

10. Versioning & Change Policy

Versi API ada di URL path (/api/v1/...). Kebijakan perubahan:

  • Breaking change (hapus/rename field, ubah tipe data, ubah semantik filter) → rilis versi baru (/v2), versi lama tetap berjalan selama masa transisi
  • Non-breaking change (tambah field baru yang opsional, tambah endpoint baru) → tidak mengubah versi, aman untuk konsumen existing
  • Masa transisi/deprecation — versi lama tidak langsung dimatikan; notifikasi dan window waktu akan diberikan sebelum versi lama disunset

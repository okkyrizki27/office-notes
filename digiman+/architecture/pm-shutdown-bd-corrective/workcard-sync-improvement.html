Workcard Sync — Proposed Improvement


Dokumen: Usulan perbaikan logic sync data Workcard untuk PM Shutdown dan BD Corrective di aplikasi Mobile Digiman+

Tujuan: Konfirmasi pendekatan sebelum implementasi di BE dan FE

Tanggal: 3 Juli 2026



1. Latar Belakang


Terdapat laporan dari site bahwa proses sync workcard memakan waktu yang cukup lama, khususnya pada dua kondisi:

  • Fresh install (pertama kali login) — sync awal tidak selesai-selesai

Setelah analisis, ditemukan dua gap utama pada logic filtering data di BE yang menjadi penyebab volume data yang ditarik terlalu besar.


2. Gap yang Ditemukan


Gap 1 — Fresh Install: Sebagian Data Tidak Ada Batas Tanggal

Pada kondisi INPROGRESS tertentu, data diambil tanpa filter tanggal apapun (semua data), sehingga workcard yang sudah sangat lama ikut ditarik.


StatusKondisiFilter Saat Ini
SUBMIT—ProjectFinish >= today − 2 bulan
IN PROGRESSNotifNo terisi dan FinishBreakdown terisiFinishBreakdown >= today − 2 bulan
IN PROGRESSNotifNo terisi dan FinishBreakdown NULLSEMUA DATA
IN PROGRESSNotifNo NULLSEMUA DATA

Gap 2 — After Last Sync: LastSyncDate Terlalu Lama

Jika user tidak membuka aplikasi selama berbulan-bulan, LastSyncDate menjadi sangat jauh ke belakang. Filter ModifiedUtcDate >= LastSyncDate akan menarik delta yang hampir sebesar fresh install — bahkan tanpa batas range.


3. Usulan Perbaikan


3.1 Konfigurasi Range (Baru)

Range hari dijadikan configurable melalui table Setting di DPlanDB, sehingga dapat diubah tanpa perlu deploy ulang.


SettingCategorySettingCodeSettingName (nilai awal)
WORKCARD_DIGIMAN+LAST_DAY_RANGE30 (hari)

Seluruh filter tanggal di bawah menggunakan nilai ini sebagai N.


3.2 Fresh Install — Logic Baru


StatusKondisiField TanggalFilter
SUBMIT — ProjectFinish >= today − N hari DIPERKETAT
IN PROGRESS NotifNo terisi dan FinishBreakdown terisi FinishBreakdown >= today − N hari DIPERKETAT
IN PROGRESS NotifNo terisi dan FinishBreakdown NULL ModifiedUtcDate >= today − N hari BERUBAH
IN PROGRESS NotifNo NULL ModifiedUtcDate >= today − N hari BERUBAH
FINISH / CANCEL — — ❌ Tidak diambil

Alasan penggunaan field:
ProjectFinish dipakai untuk SUBMIT karena ModifiedUtcDate bisa jauh ke belakang (saat workcard dibuat), sementara ProjectFinish masih dalam range relevan.

ModifiedUtcDate dipakai untuk INPROGRESS tanpa target finish — sebagai proxy aktivitas terakhir workcard. Data yang tidak disentuh lebih dari N hari dianggap tidak relevan.

3.3 After Last Sync — Logic Baru


Kondisi Normal — LastSyncDate >= today − N hari

Tidak ada perubahan signifikan. Filter tetap ModifiedUtcDate >= LastSyncDate untuk semua status.


StatusFilter
SUBMITModifiedUtcDate >= LastSyncDate
IN PROGRESSModifiedUtcDate >= LastSyncDate
FINISHModifiedUtcDate >= LastSyncDate — diambil untuk remove dari device, tidak ditampilkan
CANCELModifiedUtcDate >= LastSyncDate — diambil untuk remove dari device, tidak ditampilkan

Kondisi LastSyncDate Terlalu Lama — LastSyncDate < today − N hari BARU

Jika user tidak membuka aplikasi lebih dari N hari → gunakan Fresh Install logic (full refresh, replace seluruh data lokal di device).


Mengapa full refresh dan bukan compare-and-remove?
Delta sync hanya mengembalikan data yang berubah, bukan seluruh data valid saat ini. Pendekatan compare-and-remove akan salah menghapus workcard INPROGRESS yang masih valid namun tidak dimodifikasi dalam N hari. Full refresh lebih aman dan konsisten dengan batas relevansi N hari yang sudah disepakati.

4. Poin Konfirmasi


❓ Perlu konfirmasi dari user/stakeholder:

  • Filter 30 hari terakhir akan berdampak pada data dengan status Open (SUBMIT) maupun In Progress (INPROGRESS) — apakah acceptable jika workcard dengan status tersebut yang berumur lebih dari 30 hari tidak muncul di device?
  • Apakah acceptable jika user yang tidak membuka app lebih dari 30 hari akan mendapat full refresh (data lama di device terhapus dan diganti)?
